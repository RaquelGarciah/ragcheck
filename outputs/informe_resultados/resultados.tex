\chapter{Caso de estudio: detector de alucinaciones en RAG}

El propósito de este capítulo es aplicar las técnicas y modelos descritos previamente con el objetivo de construir un detector de alucinaciones para sistemas RAG usando machine learning clásico además de presentar y comparar los resultados con los obtenidos mediante técnicas más sofisticadas y costosas computacionalmente. 

Un sistema RAG responde a una consulta en dos fases:
\begin{itemize}
  \item \textbf{Recuperación} (\emph{retrieval}): a partir de la consulta, el
        sistema busca en una base documental las partes más relevantes y los
        recupera como contexto.
  \item \textbf{Generación} (\emph{generation}): un modelo de lenguaje redacta la
        respuesta condicionado al contexto, que llamamos fuente.
\end{itemize}

Este trabajo se centra en la segunda fase. Un RAG genera una respuesta en base a una fuente y definiremos una alucinación como una respuesta que no está respaldada por la fuente. 

Así, nuestro objetivo es construir un clasificador que, dado el par (respuesta, fuente), decida si la respuesta contiene alguna alucinación, usando exclusivamente aprendizaje automático clásico y características léxicas y estadísticas, sin redes neuronales ni llamadas a LLMs. La motivación es reproducir, a coste mucho menor, lo que hoy hace un evaluador tipo \emph{LLM como juez}, con un modelo barato, interpretable y determinista.

\section{Presentación de la base de datos}

Primero presentamos con detalle la base de datos que utilizaremos en la experimentación. Trabajamos con la versión ya procesada de RAGTruth, una base de datos creada por Niu et al.\ en 2024 \cite{niu2024ragtruth} que consta de 17.790 respuestas de LLMs etiquetadas por humanos a nivel de tramo según contengan o no alucinaciones. Dichas respuestas fueron generadas por seis modelos distintos (GPT-4, GPT-3.5-turbo, Llama-2-7B/13B/70B y Mistral-7B) sobre 2.965 prompts y cubren tres tareas de distinta naturaleza:

\begin{itemize}
  \item \textbf{QA}: dada una pregunta el modelo redacta una respuesta. La fuente es texto en prosa.
  \item \textbf{Data2txt}: dado un registro estructurado el modelo redacta una descripción. La
        fuente son campos estructurados.
  \item \textbf{Summary}: dado un documento el
        modelo produce un resumen. La fuente es un documento largo.
\end{itemize}

Se trabajará con la partición oficial de RAGTruth: $15\,090$ respuestas de entrenamiento y
$2\,700$ de test para poder comparar con el estado del arte.

La base inicialmente está formada por 11 variables entre las que se incluyen la consulta, la calidad de respuesta y la temperatura de muestreo con la que se generó; aunque nosotros solo haremos uso de las cinco siguientes.
\begin{itemize}
  \item \textbf{Context}: Contexto que debe respaldar la respuesta.
  \item \textbf{Output}: Respuesta generada por el modelo de lenguaje
  \item \textbf{Task type}: 
  \begin{itemize}
    \item \textbf{QA} 
    \item \textbf{Data2txt} 
    \item \textbf{Summary} 
  \end{itemize}
  \item \textbf{Model}: Modelo de lenguaje que genera la respuesta.
  \begin{itemize}
    \item \textbf{GPT-4} 
    \item \textbf{GPT-3.5-turbo} 
    \item \textbf{Llama-2-7B} 
    \item \textbf{Llama-2-13B} 
    \item \textbf{Llama-2-70B} 
    \item \textbf{Mistral-7B} 
  \end{itemize}
  \item \textbf{Hallucination labels}: Lista de tramos alucinados con su correspondiente tipo de alucinación. Anotadas por un humano.
\end{itemize}

Esta última variable marca las alucinaciones. A partir de ella se define la variable objetivo
binaria a nivel de respuesta: se etiqueta \tc{label}${}=1$ (alucina) si la respuesta
contiene al menos un tramo anotado, y \tc{label}${}=0$ (limpia) en caso contrario. Sobre esta
etiqueta se define la \emph{prevalencia}: la fracción de respuestas con alucinación
(\tc{label}${}=1$) en un conjunto dado. Es un término que se usará a lo largo del capítulo,
porque varía mucho de una tarea a otra y es lo que da forma al desbalance de clases. El
Cuadro~\ref{tab:muestra} muestra las primeras filas de la base de datos.

\subsubsection{Por qué un detector léxico tiene sentido}

La anatomía del tramo explica por qué un detector léxico tiene sentido: si casi todas las
alucinaciones son números, fechas o nombres propios ausentes en la fuente, medir cuánto del
vocabulario de la respuesta aparece en ella basta para delatarlas. El Cuadro~\ref{tab:ejemplo}
da el caso canónico: una cifra que nuestro detector marca al instante y que un LLM-juez deja
pasar.

Todo este análisis de los datos nos lleva a pensar que un detector léxico tiene sentido. Si casi todas las
alucinaciones son números, fechas o nombres propios ausentes en la fuente, medir cuánto del
vocabulario del output no aparece en el contexto podría aportar información. El Cuadro~\ref{tab:ejemplo} muestra cómo el ejemplo canónico es capturado por nuestro modelo mientras que un LLM es incapaz de reconocerlo. Esto ocurre por el conocido como sesgo de racionalización propio de modelos de lenguaje. La respuesta inventa la valoración $4{,}5$, que
no está en la fuente. El LLM la evalúa por plausibilidad y la racionaliza. El detector
clásico la marca por pertenencia a conjuntos

\section{Ingeniería de características}
\label{sec:fe}
Dado que los datos a trabajar están en formato de texto, el primer paso es transformar cada par (contexto, output) en un vector numérico de características que capturen indicios de alucinación. Primero se lleva a cabo un preprocesado, a continuación definimos las características candidatas,
comprobamos si separan bien las clases alucina (1), limpia (0) y finalmente seleccionamos el subconjunto óptimo con un método de selección de características.

\subsection{Preprocesado: tokenización}

Toda característica se construye sobre una tokenización simple: el texto se pasa a
minúsculas y se extraen las secuencias alfanuméricas como \emph{tokens}. Para las
características numéricas se usa una variante que conserva los decimales como una sola pieza
(``26,2'' es un token), necesaria para los $n$-gramas y la consistencia numérica. Sobre
esta tokenización se definen los dos conjuntos básicos.

\begin{definicion}
Dada una respuesta y su fuente, sea $R$ el conjunto de palabras únicas de la respuesta y
$F$ el conjunto de palabras únicas de la fuente.
\end{definicion}

$R$ y $F$ son la base de casi todas las características. La idea es que una
respuesta limpia reutiliza el vocabulario de la fuente, mientras que una alucinación
introduce términos nuevos o recombina los existentes de forma errónea.

\subsection{Características candidatas}
\label{sec:catalogo}

Se construyen dieciocho características candidatas (Cuadro~\ref{tab:feats}), agrupadas en cuatro familias. 
\begin{itemize}
  \item \textbf{Solape léxico}: mide cuánto vocabulario comparte la respuesta con la fuente.
  \item \textbf{Numérica y relacional}: busca cifras y unidades inventadas.
  \item \textbf{Nivel frase}: agrega el soporte frase a frase. Una alucinación suele ser
        una frase sin respaldo que el promedio global tapa, así para delatar la peor frase agregamos con el
        mínimo.
  \item \textbf{Codificación one-hot de la tarea}: convierte la variable de tarea, que
        toma tres valores (QA, Data2txt, Summary), en tres columnas binarias, una por tarea,
        que valen 1 en la tarea correspondiente y 0 en las demás. Así el modelo puede dar un peso
        distinto a cada tarea y especializar su decisión.
\end{itemize}

\begin{table}[H]
  \centering
  \small
  \begin{tabular}{lp{5.1cm}p{5.0cm}}
  \toprule
  Característica & Definición & Señal que captura \\

  \label{tab:feats}

Se definen en más detalle las cinco básicas de las que se deriva el resto.

\paragraph{Containment (precisión léxica).}
\begin{equation}
\text{containment} = \frac{|R \cap F|}{|R|}.
\end{equation}
Fracción de palabras distintas de la respuesta que también están en la fuente. Alto
significa que todo lo que afirma la respuesta tiene respaldo léxico; bajo, que introduce
palabras nuevas. Mide cuánto de respaldada está la respuesta.

\paragraph{Jaccard.}
\begin{equation}
\text{jaccard} = \frac{|R \cap F|}{|R \cup F|}.
\end{equation}
Solape simétrico: mismo numerador que \tc{containment} pero dividido por la unión, así
penaliza también el vocabulario de la fuente que la respuesta no usa.

\paragraph{Coseno TF-IDF.} El coseno entre los vectores TF-IDF de
respuesta y fuente,
\begin{equation}
\cos(u,v)=\frac{\sum_t u_t v_t}{\sqrt{\sum_t u_t^2}\sqrt{\sum_t v_t^2}},
\qquad u_t = \text{tf}(t)\cdot\log\frac{N}{\text{df}(t)},
\end{equation}
que pondera cada término por su rareza: las palabras comunes pesan poco y las específicas,
mucho. Mide similitud distribucional, no solo solape de conjuntos.

\paragraph{Solape numérico.}
\begin{equation}
\text{num\_overlap} = \frac{|N_R \cap N_F|}{|N_R|}\quad(=1 \text{ si } N_R=\varnothing),
\end{equation}
con $N_R, N_F$ los conjuntos de números de respuesta y fuente. Caza fechas y cifras
inventadas; si la respuesta no tiene números, no hay nada que pueda estar sin respaldo y
vale $1$.

\paragraph{Longitud de la respuesta.} \tc{answer\_len}${}=|\text{tokens}(R)|$, el número de
palabras de la respuesta (con repeticiones). Aunque no indica alucinación por sí sola ayuda a
contextualizar a las demás, por ejemplo, un containment de $0{,}8$ no significa lo mismo en 3 palabras que
en 40.


\subsection{Separabilidad de clases y correlación de variables}
\label{sec:corr}

Los modelos de machine learning ponderan más las variables que más información explican y descartan el resto. Un número elevado de variables redundantes solo añade ruido, por eso, antes de seleccionar se comprueban dos cosas: si las características discriminan y cuánta
información independiente aportan. 
Para el primer punto estudiamos la distribución de
cada característica continua separada por la variable objetivo. En variables como
\tc{containment}, \tc{jaccard} o \tc{num\_context} la clase alucinada se desplaza hacia
menos solape, como se aprecia en la Figura~\ref{fig:sep}.

\begin{figure}[H]
  \centering
  \includegraphics[width=\figw]{eda_separabilidad_features}
  \caption{Distribución de cada característica continua separada por clase.}
  \label{fig:sep}
  \end{figure}

Discriminar no es suficiente: dos características muy correlacionadas dicen lo mismo. La
Figura~\ref{fig:corr} muestra la correlación de Spearman entre las dieciocho. El bloque de solape
léxico está muy correlacionado ($|\rho|>0{,}8$). \tc{containment} correlaciona con
\tc{tfidf\_cos}, \tc{sent\_cont\_mean}, \tc{sent\_sim\_mean}, \tc{novel\_bigram} y
\tc{novel\_trigram}, y estas dos últimas son casi el negativo exacto de \tc{containment}.
Lo que nos sugiere que sobran variables y las dieciocho carácterísticas aportan información en cuatro ejes ortogonales distintos:

\begin{enumerate}
  \item \textbf{Soporte léxico}: containment, jaccard, tfidf\_cos, novedad de $n$-gramas,
        agregados por frase. Todas miden cuánto se apoya la respuesta en la
        fuente.
  \item \textbf{Consistencia numérica}: \tc{num\_overlap} y \tc{num\_context}, poco correlacionadas con el eje anterior.
  \item \textbf{Longitud}: \tc{answer\_len}, ortogonal al resto.
  \item \textbf{Polaridad}: \tc{neg\_diff}, señal débil pero independiente.
\end{enumerate}

\subsection{Selección de variables mediante Recursive Feature Elimination}
\label{sec:rfe}
\subsubsection{Métodos de selección: filtro y envolvente}
Los métodos de selección de características se dividen en dos grandes familias
\cite{guyon2003selection}.
\begin{itemize}
  \item \textbf{Métodos de filtro}: puntúan cada característica por una medida estadística de
        su relación con la etiqueta (correlación, información mutua, $\chi^2$) y son independientes al modelo utilizado. Son rápidos, pero ignoran las interacciones y la
        redundancia entre variables.
  \item \textbf{Métodos envolventes} (\emph{wrapper}): usan el propio modelo como juez. Prueban
        subconjuntos, los entrenan y se quedan con el que mejor rendimiento obtiene. Son más caros pero más precisos.
\end{itemize}

Como aquí el problema es justamente la redundancia (sección~\ref{sec:corr}), usamos un método
envolvente.

\subsubsection{El método RFE}

\emph{Recursive Feature Elimination} (RFE) \cite{guyon2002rfe} es el método envolvente
que usaremos. Entrena el modelo con todas las características, se elimina la menos relevante
según la importancia interna del modelo, se reentrena y se repite hasta agotar las
variables. El orden inverso de eliminación produce un ranking de más a menos
importancia.

\section{Metodología de entrenamiento y validación}

Esta sección describe el flujo de trabajo desde la base de datos en crudo hasta la evaluación final. El siguiente esquema muestra el procedimiento seguido.

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{concept_pipeline}
\caption{Flujo de trabajo: del dato en crudo a la evaluación en el test.}
\label{fig:pipeline}
\end{figure}


Para empezar partimos la base en los dos bloques:
\begin{itemize}
  \item \textbf{Entrenamiento} ($15\,090$ respuestas): subconjunto sobre el que se ajustan las
        características, los hiperparámetros y el umbral, con validación cruzada interna.
  \item \textbf{Test} ($2\,700$ respuestas): subconjunto reservado para la evaluación final.
\end{itemize}

Seguir la partición natural de RAGTruth nos permite comparar con el estado del arte sobre el mismo test, y
evaluarlo una sola vez evita el sobreajuste por iteración, un riesgo al probar muchas
configuraciones. Sin embargo, reservar el test no es suficiente. Al entrenar y validar sobre el
mismo bloque surgen tres problemas que abordamos en el resto de la sección:
\begin{itemize}
  \item Los resultados dependen de las particiones en el entrenamiento.
  \item Los hiperparámetros pueden ajustarse a una partición concreta.
  \item El desbalance de clases cambia de una tarea a otra.
\end{itemize}
A continuación trataremos cada problema por separado.

\subsection{Problema 1: dependencia resultados-partición}
\label{sec:gkf}

Para no depender de una única partición se usa validación cruzada: en lugar de un solo corte
entrenamiento-validación, se divide el entrenamiento en $k$ partes (\emph{folds}) y se repite
el ajuste $k$ veces, dejando cada vez una parte como validación y las otras $k-1$ como
entrenamiento. La métrica final será la media de las $k$ iteraciones, más estable que la de un
único corte. Usaremos $k=6$.

En nuestra base de datos, además, varias respuestas comparten el mismo documento como fuente, y si un documento
aparece a la vez en entrenamiento y validación el modelo memoriza en lugar de generalizar e
infla el resultado. Por eso se agrupa por \tc{source} con \emph{GroupKFold}: cada documento
cae entero en un solo fold (Figura~\ref{fig:gkf}), sin repartirse entre entrenamiento y
validación. Un KFold aleatorio, en cambio, filtraría información entre folds.

\begin{figure}[H]
\centering
\includegraphics[width=0.72\textwidth]{concept_groupkfold}
\caption{GroupKFold por documento: cada \tc{source} cae entero en un fold, sin fuga entre
entrenamiento y validación.}
\label{fig:gkf}
\end{figure}

\subsection{Problema 2: dependencia hiperparámetros-entrenamiento}

Los modelos con hiperparámetros pueden sobreajustarlos. Para no caer en el \emph{overfitting} los elegimos usando \emph{grid
search}: se define una rejilla de combinaciones de hiperparámetros, se evalúa cada una por
validación cruzada GroupKFold sin tocar el test y se conserva la que mejor resultado da, en nuestro caso, la combinación con mejor F1.(Figura~\ref{fig:grid}). 

\begin{figure}[H]
\centering
\includegraphics[width=0.6\textwidth]{concept_gridsearch}
\caption{Búsqueda en rejilla: cada celda es una combinación de hiperparámetros, evaluada por la
media de F1 de seis folds; se elige la de mayor F1 de validación.}
\label{fig:grid}
\end{figure}

El coste es multiplicativo: optimizar tres hiperparámetros con cuatro valores cada uno y seis
folds exige $4\cdot4\cdot4\cdot6=384$ ajustes del modelo. Ese coste, asumible en CPU para
modelos clásicos, es una de las ventajas del enfoque frente a las redes profundas,
donde cada ajuste cuesta horas de GPU.

\subsection{Problema 3: desbalance y umbral de decisión}
\label{sec:desb-trat}

El tercer problema es propio del desbalance de nuestra base. El modelo produce un \emph{score} continuo por
respuesta, y el \textbf{umbral} es el corte que lo convierte en decisión: se considera alucinación si el score lo supera. Moverlo desliza el trade-off entre precisión y exhaustividad, bajarlo caza más alucinaciones a
costa de más falsas alarmas (Figura~\ref{fig:umbral-esq}a). Como la prevalencia de alucinaciones difiere por tarea, el corte óptimo es distinto en cada una: donde la alucinación es rara
(QA, Summary) conviene bajar el umbral para no dejarla pasar (Figura~\ref{fig:umbral-esq}b).

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{concept_umbral}
\caption{(a) El umbral separa las dos clases sobre el score; moverlo intercambia falsos negativos
por falsos positivos. (b) Cada tarea opera a una prevalencia distinta, así que su corte óptimo
también es distinto.}
\label{fig:umbral-esq}
\end{figure}

La Figura~\ref{fig:umbral} compara tres criterios para fijar el umbral, todos ajustados en entrenamiento y aplicados al test: el índice de Youden global, el máximo de F1 por tarea y el máximo de $F_2$ por tarea. Si fijamos el umbral por tarea recuperamos falsos
negativos que el umbral global deja pasar, a costa de perder precisión. Pasamos de 285 falsos negativos a 125.

\subsubsection{Regresión logística}

La regresión logística modela la probabilidad de alucinación aplicando la función sigmoide
$\sigma(z)=1/(1+e^{-z})$ a una combinación lineal de las once características.
Es el modelo más interpretable de todos, ya que cada coeficiente es el
peso, con su signo, de una característica, y también el más barato de entrenar; su límite es
que solo puede trazar fronteras de decisión lineales. Se ajustan tres hiperparámetros (con el
solucionador fijado en \tc{liblinear}, que admite las dos penalizaciones):

\begin{itemize}
\item \tc{C}: cuanto menor es \tc{C}, más se
penalizan los coeficientes grandes y más simple resulta el modelo, lo que reduce el
sobreajuste. Se prueban los valores $0{,}01$, $0{,}1$, $1$, $10$ y $100$.
\item \tc{penalty}: el tipo de penalización sobre los coeficientes.
  \begin{itemize}
  \item \tc{l1} (Lasso): tiende a anular coeficientes, seleccionando características.
  \item \tc{l2} (Ridge): encoge todos los coeficientes sin anularlos.
  \end{itemize}
\item \tc{class\_weight}: determina cómo pesa cada clase en el ajuste.
  \begin{itemize}
  \item \tc{None}: todas las respuestas pesan igual.
  \item \tc{balanced}: cada clase se pondera de forma inversamente proporcional a su
  frecuencia, de modo que la clase minoritaria (la alucinación) pesa más. Es una manera de
  compensar el desbalance directamente en el entrenamiento.
  \end{itemize}
\end{itemize}

En validación cruzada, la mejor combinación es \tc{C}${}=0{,}01$,
\tc{penalty}${}=$\tc{l2} y \tc{class\_weight}${}=$\tc{balanced} (Cuadro~\ref{tab:grid-lr}); lo
que marca la diferencia no es el valor de \tc{C} ni la penalización, sino activar el
reponderado de clases. Aplicando esa configuración sobre el test se obtienen los resultados del
Cuadro~\ref{tab:met-lr}. Es el modelo más débil: su AUC ($0{,}798$) es el
único que no llega a $0{,}80$, coherente con que solo puede trazar fronteras lineales.

\subsubsection{K vecinos más cercanos (KNN)}

El algoritmo de los $k$ vecinos más cercanos clasifica cada respuesta según la clase
mayoritaria entre las $k$ del entrenamiento más parecidas a ella. No
asume ninguna forma para la frontera de decisión, pero es sensible a la escala de las
características, por eso se estandarizan antes, y su inferencia es costosa, ya que guarda todo
el conjunto de entrenamiento. Se ajustan tres hiperparámetros:

\begin{itemize}
\item \tc{n\_neighbors} ($k$): número de vecinos que se consultan para decidir la clase. Un
$k$ pequeño da fronteras muy locales y ruidosas; uno grande, fronteras más suaves. Se prueban
los valores $3$, $5$, $11$, $15$ y $25$.
\item \tc{weights}: cómo se combinan los votos de los $k$ vecinos.
  \begin{itemize}
  \item \tc{uniform}: todos los vecinos pesan igual.
  \item \tc{distance}: cada vecino pesa de forma inversa a su distancia, de modo que los más
  cercanos influyen más en la predicción.
  \end{itemize}
\item \tc{p}: el orden de la distancia de Minkowski con que se miden los vecinos.
  \begin{itemize}
  \item \tc{p=1}: distancia Manhattan (suma de diferencias absolutas).
  \item \tc{p=2}: distancia euclídea.
  \end{itemize}
\end{itemize}

La rejilla elige $k=25$, distancia
\tc{Manhattan} ($p=1$) y ponderación por distancia; la distancia Manhattan supera a la
euclídea de forma sistemática, y el rendimiento se satura a partir de $k\approx15$. Sobre el
test separa razonablemente bien
(AUC $0{,}826$), aunque no destaca en precisión ni en recall.

\subsubsection{Máquina de vectores de soporte (SVM)}

La máquina de vectores de soporte busca la frontera que separa las dos clases con el mayor
margen posible. Con un núcleo de base radial (RBF) esa frontera puede ser no lineal, porque el
núcleo proyecta las características a un espacio de mayor dimensión. Da
buenas fronteras curvas con control del sobreajuste, dice de qué lado del margen cae un nuevo registro, pero no una probabilidad, para obtenerla se calibra aparte con Platt scaling, un ajuste por validación cruzada y su coste crece con el cuadrado del número de ejemplos, por lo
que la búsqueda de hiperparámetros se hace sobre una submuestra. Las características se
estandarizan antes porque la sensibilidad del modelo a las distancias. Se ajustan tres hiperparámetros:

\begin{itemize}
\item \tc{C}: regula el compromiso entre un margen ancho y clasificar bien el entrenamiento.
Un \tc{C} pequeño tolera más errores (margen ancho, modelo más simple). Uno grande penaliza
cada error (margen estrecho). Se prueban los valores $0{,}1$, $1$, $10$ y $100$.
\item \tc{gamma}: anchura del núcleo RBF, es decir, hasta qué distancia influye cada ejemplo.
  \begin{itemize}
  \item \tc{scale}: usa $1/(n_\text{car}\cdot \operatorname{Var}(X))$, que tiene en cuenta la
  dispersión de los datos.
  \item \tc{auto}: usa $1/n_\text{car}$.
  \end{itemize}
\item \tc{class\_weight}: \tc{None} (clases al mismo peso) o \tc{balanced} (la clase
minoritaria pesa más), para compensar el desbalance como en la regresión logística.
\end{itemize}

El mejor punto de la rejilla es \tc{C}${}=10$,
\tc{gamma}${}=$\tc{scale} y \tc{class\_weight}${}=$\tc{balanced}; \tc{gamma} no influye y, como
en la regresión logística, el reponderado de clases es lo que ayuda. Sobre el test rinde de
forma sólida: AUC $0{,}819$ y F1 $0{,}685$, empatado con XGBoost en F1, aunque queda tercero por
promedio F1--AUC, tras XGBoost y el bosque aleatorio.

\subsubsection{Bosque aleatorio}

El bosque aleatorio promedia el voto de muchos árboles de decisión, cada uno entrenado sobre
una submuestra distinta de los datos y de las características. Es robusto,
capta interacciones entre características y es insensible a la escala, a cambio de perder la
interpretabilidad de un árbol único. Se fija el número de árboles en $300$, suficiente para
que el promedio se estabilice, y se ajustan tres hiperparámetros:

\begin{itemize}
\item \tc{max\_depth}: profundidad máxima de cada árbol, es decir, cuántas preguntas
encadenadas puede hacer. Árboles poco profundos capturan patrones gruesos; muy profundos,
detalles finos a riesgo de sobreajustar. Se prueban $10$, $15$, $20$ y \tc{None} (sin límite:
el árbol crece hasta separar el entrenamiento).
\item \tc{max\_features}: cuántas características se consideran, elegidas al azar, en cada
división de un árbol. Reducirlas descorrelaciona los árboles entre sí y suele mejorar el
promedio.
  \begin{itemize}
  \item \tc{sqrt}: la raíz cuadrada del total (aquí $\approx 3$ de $11$).
  \item \tc{log2}: el logaritmo en base dos del total.
  \end{itemize}
\item \tc{min\_samples\_leaf}: número mínimo de muestras que debe quedar en una hoja. Valores
mayores producen hojas más pobladas y árboles más suaves, lo que regulariza. Se prueban $1$,
$2$, $4$ y $8$.
\end{itemize}

La búsqueda se queda con profundidad máxima $20$,
\tc{max\_features}${}=$\tc{sqrt} y \tc{min\_samples\_leaf}${}=4$, pero las $32$ combinaciones
rinden casi igual (entre $0{,}688$ y $0{,}695$): el modelo es muy robusto a sus
hiperparámetros. Sobre el test alcanza un AUC de $0{,}833$, de los
más altos.

\subsubsection{Gradient boosting (XGBoost)}

El \emph{gradient boosting} construye los árboles de forma secuencial: cada árbol nuevo se
entrena para corregir los errores que comete la suma de los anteriores. Suele ser el mejor clasificador en datos tabulares y
produce probabilidades bien calibradas, a cambio de tener más hiperparámetros que ajustar. Se
fijan el número de árboles ($300$), el submuestreo ($0{,}8$) y la regularización, y se ajustan
los dos hiperparámetros más influyentes:

\begin{itemize}
\item \tc{max\_depth}: profundidad de cada árbol; controla la complejidad de las interacciones
que puede capturar. Se prueban los valores $3$, $4$, $5$ y $6$.
\item \tc{learning\_rate}: tasa de aprendizaje, es decir, cuánto contribuye cada árbol nuevo a
la predicción. Una tasa baja aprende más despacio pero generaliza mejor. Se prueban los
valores $0{,}03$, $0{,}05$, $0{,}1$ y $0{,}2$.
\item \tc{subsample}: fracción de ejemplos, tomada al azar, con la que se entrena cada árbol.
Menos del total ($0{,}8$) introduce aleatoriedad y regulariza; $1{,}0$ usa todos. Se prueban
$0{,}8$ y $1{,}0$.
\end{itemize}

Gana la combinación profundidad $6$, tasa de
aprendizaje $0{,}05$ y \tc{subsample}${}=0{,}8$, pero las $32$ combinaciones rinden muy
parecido (entre $0{,}678$ y $0{,}696$). Sobre el test obtiene el
mejor AUC de los cinco modelos, $0{,}837$.

\section{Comparación con el estado del arte}
\label{sec:sota}

\subsection{Los métodos de referencia}

El estado del arte sobre RAGTruth es enteramente neuronal y se agrupa en tres familias:
\begin{itemize}
  \item \textbf{LLM-juez} (GPT-4-turbo, GPT-4o): un modelo de lenguaje grande emite el veredicto
  a partir de un \emph{prompt}, sin entrenarse. No es determinista y cuesta del orden de un
  céntimo y varios segundos por evaluación.
  \item \textbf{Detectores \emph{fine-tuned}} (Llama-2-13B, Lynx-8B/70B, RAG-HAT): ajustan un LLM
  completo sobre RAGTruth. Máximo rendimiento, a cambio de miles de millones de parámetros y GPU.
  \item \textbf{Encoders ligeros} (Luna, LettuceDetect, $150$--$396$\,M sobre ModernBERT): un
  punto intermedio, también neuronal y sobre GPU.
\end{itemize}

Frente a miles de millones de parámetros y GPU, nuestro modelo clásico usa del orden de $10^{4}$ parámetros (unos $300$ árboles someros sobre $11$ características), corre en CPU y es determinista.
